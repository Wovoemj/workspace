## 景点加载慢修复（app.py）- 重大性能优化
### 根本原因
Redis 不可用时，`get_redis()` 每次请求都会重新建立 TCP 连接并等待超时（socket_connect_timeout=5s × 3次操作 = 15-18s）

### 修复方案
1. `get_redis()` 新增 `_redis_available` 布尔标记，Redis 不可用后**永久标记**，不再重试连接
2. `socket_connect_timeout` 从 5s 降至 0.5s，快速失败
3. 添加进程内内存缓存 `_mem_cache`（dict），Redis 不可用时自动降级到内存缓存
4. `redis_cache_get/set` 改为双层：先 Redis，失败后用内存缓存
5. `popular` 排序去掉 CASE WHEN（1452 次全表扫描），改为 `rating DESC`

### 效果
- 修复前：每次请求 18s（Redis 超时叠加）
- 修复后：首次 ~3s（冷启动 + Redis 检测一次），之后 0.01s（内存缓存命中）

## 周边联动假数据 → 真实 POI（NearbyMap + app.py）
### 问题
`/api/nearby` 接口使用 `generate_nearby_pois()` 随机生成假的周边 POI（万达广场、便利蜂、人民路等），珠穆朗玛峰旁边也会出现这些城市设施，数据完全虚构。

### 修复方案
1. **删除** 后端 `generate_nearby_pois()` 假数据生成函数
2. **前端** NearbyMap 组件改用高德 AMap.PlaceSearch 插件直接搜索周边真实 POI
3. 后端 `/api/nearby` 只返回数据库中附近的真实景点
4. 高德 Web 服务 API 需要「Web服务」类型的 Key，当前 Key（ec38e...）是 JS SDK 类型（USERKEY_PLAT_NOMATCH），所以改用前端 JS SDK 的 PlaceSearch 插件绕过
5. 偏远地区（如珠穆朗玛峰）没有 POI 时不显示假数据，直接显示"暂无周边数据"

### 改动文件
- `app.py`: 删除 `generate_nearby_pois()`、`fetch_amap_nearby_pois()`，`/api/nearby` 不再生成假 POI
- `frontend/user-web/src/components/NearbyMap.tsx`: 加载 `AMap.PlaceSearch` 插件，用 `searchNearBy` 搜索真实 POI
- `.env`: 新增 `AMAP_API_KEY`（后端用，暂时无效因为 Key 类型不匹配）
